# Supabase nested-join type literals duplicated across 4+ scripts

## Problem Statement
The nested `restaurant -> park` join shape is hand-written as a type literal in at least four scripts (`estimate-nutrition-codex.ts`, `estimate-nutrition-ai-nameonly.ts`, `insert-allears-disney.ts`, `upgrade-chain-nutrition.ts`). Any drift in column names or nullability has to be patched in N places, and TypeScript can't catch it. Extracting a shared `RestaurantJoin` / `ParkJoin` interface eliminates the drift class entirely.

## Findings
- **Source agent:** kieran-typescript-reviewer
- **Evidence:** `scripts/estimate-nutrition-codex.ts:46-52`, `scripts/estimate-nutrition-ai-nameonly.ts:98-100`, `scripts/insert-allears-disney.ts`, `scripts/upgrade-chain-nutrition.ts:301`.
- **Severity rationale:** P2 — no runtime bug today, but a refactor hazard once we evolve the join shape.

## Proposed Solutions

### Option A — Extract shared types module (recommended)
- **What:** Create `scripts/lib/supabase-joins.ts` exporting `ParkJoin`, `RestaurantJoin`, `MenuItemWithRestaurant` types and a typed `select()` string constant.
- **Pros:** Single source of truth; rename refactors flow automatically.
- **Pros:** Forces consistent join column lists across scripts.
- **Cons:** New file to maintain.
- **Cons:** Touching ~4-5 scripts in one PR.
- **Effort:** Small
- **Risk:** Low

### Option B — Use `database.types.ts` generated by supabase CLI
- **What:** Regenerate the typed schema and import `Database['public']['Tables']['restaurants']['Row']` etc.
- **Pros:** Authoritative against the live schema.
- **Pros:** Catches schema drift automatically.
- **Cons:** Requires Supabase CLI typing pipeline to be set up + committed.
- **Cons:** Nested-relationship shapes still need a hand-written wrapper.
- **Effort:** Medium
- **Risk:** Low
